## Bug 1: Foldseek Multihit 모드에서 `-m aligned` 적용 시 superposition이 안 되는 문제

### 현상
Foldseek multihit 모드에서 `-m aligned`를 설정하고 hit를 탐색할 때, query 단백질의 `is_aligned` 플래그가 이전 hit의 값을 그대로 유지하여 정렬 영역이 올바르게 표시되지 않는다.

### 근본 원인
`compute_aligned_regions_from_aln()` (`Protein.cpp:601-660`)에서 정렬된 잔기에 대해 `is_aligned = true`만 설정하고, **비정렬 잔기에 대해 `is_aligned = false`로 초기화하는 코드가 없다.**

`load_next_hit()` (`Screen.cpp:1324-1611`)에서 새 hit를 로드할 때:
- 기존 target(data[1])은 삭제 후 새로 생성 → `is_aligned = false` (기본값)
- **query(data[0])는 삭제되지 않음** → 이전 hit에서 설정된 `is_aligned = true` 플래그가 잔류
- 새 hit의 `compute_aligned_regions_from_aln()` 호출 시 새 정렬에 해당하는 잔기만 `true`로 설정
- 이전 hit에서 정렬되었지만 현재 hit에서는 비정렬인 잔기가 여전히 `true`로 남음

### 관련 코드

| 파일 | 라인 | 내용 |
|------|------|------|
| `src/structure/Protein.cpp` | 601-660 | `compute_aligned_regions_from_aln()` — `is_aligned = true`만 설정, false 초기화 없음 |
| `src/structure/Protein.cpp` | 478-530 | `sync_bool_field_to_screen()` — init_atoms → screen_atoms 동기화 |
| `src/structure/Protein.cpp` | 536-538 | `sync_aligned_to_screen()` — 위 함수의 wrapper |
| `src/visualization/Screen.cpp` | 1360-1368 | `load_next_hit()` — 기존 target 삭제, query(data[0])는 유지 |
| `src/visualization/Screen.cpp` | 1600-1608 | `load_next_hit()` — aligned 모드 시 `compute_aligned_regions_from_aln()` 호출 |

### 수정 방안

**방안: `compute_aligned_regions_from_aln()` 진입 시 양쪽 단백질의 `is_aligned`를 전부 false로 초기화**

**수정 파일: `src/structure/Protein.cpp`**

`compute_aligned_regions_from_aln()` 함수 시작 부분 (라인 601 이후, 본격적인 정렬 로직 전)에 다음 초기화 코드를 추가:

```cpp
void Protein::compute_aligned_regions_from_aln(Protein& other, ...) {
    if (qaln.size() != taln.size()) return;
    float thr2 = threshold * threshold;

    // === 추가: is_aligned 플래그 초기화 ===
    for (auto& [cid, chain] : init_atoms) {
        for (Atom& a : chain) {
            a.is_aligned = false;
        }
    }
    for (auto& [cid, chain] : other.init_atoms) {
        for (Atom& a : chain) {
            a.is_aligned = false;
        }
    }
    // === 초기화 끝 ===

    // 기존 로직 계속...
    std::vector<Atom*> query_cas;
    // ...
}
```

이렇게 하면 매 hit 전환 시 query와 target 모두 `is_aligned`가 깨끗하게 초기화되어, 현재 hit의 정렬 정보만 반영된다.

### 테스트 시나리오
1. `./StrucTTY example/1NPL-assembly1.cif -fs example/foldseek_result/alis_pdb100.m8 -m aligned`
2. 여러 hit를 순차 탐색하면서 query의 정렬 영역이 hit마다 정확히 달라지는지 확인
3. 이전 hit의 정렬 영역이 다음 hit에서 잔류하지 않는지 확인

## Bug 2: Conservation 모드에서 homodimer의 첫 번째 chain만 시각화되는 문제

### 현상
Homodimer(동일 서열의 체인 A, B를 가진 단백질)에서 conservation 모드를 사용하면, 첫 번째 체인(A)에만 conservation 색상이 적용되고 나머지 체인(B)은 olive dim (color 11)으로 표시되어 사실상 보이지 않는다.

### 근본 원인
`apply_conservation_scores()` (`Protein.cpp:726-740`)에서 conservation 점수 벡터를 **모든 체인에 걸쳐 순차적으로(flat하게)** 적용한다. 그런데 MSA에서 계산된 점수 벡터의 길이는 query 서열(단일 체인) 길이뿐이므로, 두 번째 체인부터는 `idx >= score_len` 조건에 걸려 점수가 할당되지 않는다.

```
체인 A: 잔기 0~99 → scores[0]~scores[99] 적용됨 ✓
체인 B: 잔기 100~199 → idx(100~199) >= score_len(100) → 스킵됨 ✗
```

### 관련 코드

| 파일 | 라인 | 내용 |
|------|------|------|
| `src/structure/Protein.cpp` | 726-740 | `apply_conservation_scores()` — 순차 인덱스로 scores 적용, 벡터 길이 초과 시 스킵 |
| `src/structure/MSAParser.cpp` | 78-79 | `load()` — query_sequence = sequences[0]만 사용 |
| `src/structure/MSAParser.cpp` | 82-127 | `compute_conservation()` — query_len 크기 벡터만 반환 |
| `src/structure/FoldMasonParser.cpp` | 227-235 | `build_query_col_map()` — query entry의 non-gap 위치만 매핑 |
| `src/structty.cpp` | 118-126 | MSA 기반 conservation 초기화 — protein index 0에만 적용 |
| `src/structty.cpp` | 178-188 | FoldMason 기반 conservation 초기화 — 동일 문제 |
| `src/visualization/Screen.cpp` | 551-566 | conservation 렌더링 — score < 0이면 olive dim (color 11) |

### 수정 방안

**방안: `apply_conservation_scores()`에서 체인별로 점수를 반복 적용 (homodimer 대응)**

**수정 파일: `src/structure/Protein.cpp`**

현재 `apply_conservation_scores()`의 flat 인덱싱 방식을 체인별 반복 인덱싱으로 변경한다. Homodimer의 경우 각 체인이 동일 서열이므로, 점수 벡터를 체인마다 처음부터 다시 적용하면 된다.

```cpp
void Protein::apply_conservation_scores(const std::vector<float>& scores) {
    int score_len = (int)scores.size();
    if (score_len == 0) return;

    // 전체 잔기 수 계산
    int total_residues = 0;
    for (auto& [cid, chain] : init_atoms) {
        total_residues += (int)chain.size();
    }

    if (total_residues <= score_len) {
        // 기존 방식: scores가 전체 잔기를 커버하면 순차 적용
        int idx = 0;
        for (auto& [cid, chain] : init_atoms) {
            for (Atom& a : chain) {
                if (idx < score_len) {
                    a.conservation_score = scores[idx];
                }
                idx++;
            }
        }
    } else {
        // Homodimer 대응: scores가 전체 잔기보다 짧으면 체인별로 반복 적용
        for (auto& [cid, chain] : init_atoms) {
            int idx = 0;
            for (Atom& a : chain) {
                if (idx < score_len) {
                    a.conservation_score = scores[idx];
                }
                idx++;
            }
        }
    }

    sync_conservation_to_screen();
}
```

**핵심 로직**: `total_residues > score_len`인 경우 (homodimer 등), 각 체인마다 `idx`를 0부터 다시 시작하여 동일한 conservation 점수를 반복 적용한다. Heterodimer처럼 서로 다른 체인인 경우에도 score_len이 단일 체인 길이이므로 각 체인별로 독립 매핑하는 것이 올바르다.

### 테스트 시나리오
1. Homodimer 구조 (예: 1CJK 또는 유사 homodimer) + MSA 파일로 conservation 모드 실행
2. 모든 체인에 conservation 색상이 동일하게 적용되는지 확인
3. 단일 체인 단백질에서 기존 동작이 유지되는지 확인 (regression 방지)
4. FoldMason 모드에서도 homodimer conservation이 올바르게 작동하는지 확인
